# Group 32B Media Foundation Implementation

Group 32B implements the approved media foundation needed before seller UI implementation. It keeps the existing unified `media_assets` and `media_links` architecture and adds the narrow trusted upload, management, and projection layer required for V1 seller media.

This group does not add React/UI code, Equipment & Supplies media, breed image management, video support, signed/private delivery, AI image generation, Stripe changes, or broad moderation tooling.

## Storage Model

V1 seller media uses one public Supabase Storage bucket:

- `seller-media`

The bucket is intended only for public-facing storefront and live animal listing images. Private documents, payment files, admin-only evidence, and future non-public assets should not use this bucket.

Supported upload types:

- `image/jpeg`
- `image/png`
- `image/webp`

Rejected upload types:

- `image/svg+xml`
- any other unsupported MIME type

The upload limit is 8 MB per original file. The upload Edge Function validates file type, file size, and image dimensions before creating database records.

Storage paths are generated by the trusted upload boundary, not by sellers:

```text
stores/{store_id}/images/{yyyy}/{mm}/{uuid}-{random}.{ext}
```

Sellers do not provide or update `bucket_name`, `storage_path`, `asset_status`, or `moderation_status`.

## Existing Tables Used

No new media table family is created.

Existing tables used:

- `media_assets`
- `media_links`

`media_assets` stores uploaded object metadata.

`media_links` stores where an asset appears, including:

- entity type
- entity ID
- display context
- sort order
- featured flag
- link-level alt text override
- caption
- visibility status

## Supported Entities

The media foundation supports the existing entity types:

- `store`
- `listing_batch`
- `listing_batch_breed`
- `inventory_item`
- `seller_breed_profile`

Every media operation validates that the target entity belongs to the same seller store as the media asset.

Recommended display contexts:

- Store logo: `store` + `logo`
- Store banner: `store` + `hero`
- Store gallery: `store` + `gallery`
- Listing photos: `listing_batch` + `gallery`
- Listing featured image: `listing_batch` + `primary` or featured gallery item
- Inventory-row photos: `inventory_item` + `gallery`
- Inventory-row featured image: `inventory_item` + `primary` or featured gallery item
- Seller breed profile photos: `seller_breed_profile` + `primary` or `gallery`

## Upload Boundary

Seller uploads go through:

- `supabase/functions/seller-media-upload/index.ts`

The function:

- requires an authenticated seller session
- validates the buyer-facing image type and file size
- rejects SVG
- validates dimensions where practical
- generates a store-scoped randomized storage path
- uploads to `seller-media`
- calls the service-role-only `seller_create_uploaded_media` RPC
- creates one `media_assets` row
- creates one `media_links` row
- returns a seller-safe media response
- returns fixed public-safe error codes/messages for callers
- logs raw internal upload/RPC/config errors server-side only

The Edge Function uses service role only inside the trusted server boundary. Ownership is still enforced by passing the authenticated user ID into the RPC and validating that the user owns the target store or is an admin.

## Seller Management RPCs

Seller-safe media operations are exposed through narrow RPCs:

- `seller_attach_media`
- `seller_reorder_media`
- `seller_replace_store_media`
- `seller_replace_media_link`
- `seller_archive_media_link`
- `seller_set_media_featured`
- `seller_update_media_text`

The service-role upload helper is:

- `seller_create_uploaded_media`

Sellers should not write directly to `media_assets` or `media_links` for create/update behavior. Group 32B removes the earlier broad owner insert/update RLS policies from those tables and routes mutation through RPCs.

## Ordering And Featured Images

Multiple photos are supported through multiple active `media_links` rows.

Ordering is explicit and stable:

- `media_links.sort_order`

Featured image selection is explicit:

- `media_links.is_featured`

Only one active featured image is allowed per entity and display context through the existing partial unique index. When a featured link is archived, the next active media link in sort order is promoted automatically when available.

## Replacement And Removal

Replacement uses archive/hide behavior rather than hard deletion.

Store logo/banner replacement:

- archives the existing active `logo` or `hero` link
- creates or reactivates a new active featured link
- leaves the old asset record in place

Gallery replacement:

- archives the old active link
- links the new asset in the same entity/context/sort position
- preserves featured status when replacing a featured item

Removal:

- archives the media link
- does not delete the storage object immediately
- does not hard-delete media metadata

Storage object cleanup remains a future maintenance task.

## Seller Read Projection

Seller edit screens can read:

- `seller_media_management`

It includes:

- media asset ID
- media link ID
- store ID
- entity type
- entity ID
- display context
- public URL
- alt text
- caption
- sort order
- featured flag
- moderation status
- asset status
- visibility status
- file metadata

The projection is seller/admin scoped.

## Public Gallery Projection

Public storefront/gallery rendering can read:

- `public_storefront_media_gallery`

It exposes only:

- active media links
- active media assets
- approved assets
- live stores
- visible public listing/inventory records

The projection supports ordered galleries for:

- storefront media
- listing batch media
- listing batch breed media
- inventory item media

It intentionally does not expose seller-private media management fields.

## Moderation Behavior

V1 uploads are auto-approved after basic validation.

New uploads are created with:

- `asset_status = 'active'`
- `moderation_status = 'approved'`
- `moderation_checked_at = now()`

There is no moderation queue or admin approval requirement in Group 32B. Admin takedown and advanced moderation automation remain future work.

## Security Notes

Security controls added in Group 32B:

- direct owner insert/update policies on `media_assets` and `media_links` are removed
- mutation is routed through RPCs
- upload creation is service-role-only
- seller management RPCs are authenticated-only
- helper functions use `security definer` with `set search_path = public`
- broad public execute is revoked from sensitive RPCs
- public gallery access is read-only and filtered to active approved media
- ownership is validated for stores, listing batches, listing breed rows, inventory rows, and seller breed profiles

Because `seller-media` is public, images uploaded there should be treated as public once attached to a public-facing record. Sensitive files should not be uploaded through this foundation.
